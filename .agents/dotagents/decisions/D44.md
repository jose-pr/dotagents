# D44: role-based multi-provider executor routing

Date: 2026-07-16

## Decision

The model catalog assigns provider models to named `family/subrole` jobs such as
`code-executor/implemented`, `architecture-planner/plan`, and
`reviewer/security`. Plans name only the role/subrole. The executor reads
`MODELS.md`, selects the calling agent's host-native provider lane, applies its
settings, and records the resolved provider/model. Cross-provider fallback needs
explicit permission.

The catalog covers the user's ChatGPT Plus, Claude Pro, and Gemini Pro surfaces.
OpenAI GPT-5.6 Sol/Terra/Luna are the current frontier/balanced/cost-sensitive
tier; GPT-5.3-Codex remains the Codex-specific coding route. Anthropic and Google
entries are explicit cross-provider fallbacks only when that run permits them.

## Rationale

Raw model IDs in plans make a plan brittle across subscriptions and execution
surfaces. A role contract preserves the task capability while allowing the local
Codex surface, Claude, or Gemini to execute it. Separating subscription access
from actual callability prevents the executor from assuming that an app
subscription is a local or API subagent.

## Changes

- `overlays/flows/MODELS.md` defines roles, grouped capabilities, provider models,
  settings, routing, and current official references.
- `overlays/flows/flows/PLAN.md` requires role/subrole executor lines and includes
  MODELS in the kickoff prompt.
- `overlays/flows/flows/EXEC.md` resolves and records the actual route and handles
  provider fallback without changing role families.
