Date: 2026-07-20

## Decision

`leak_check.py` splits harvested plan basenames into two tiers:

- **multi-word** (contains `_`, as the snake_case plan-naming rule requires) →
  hard failure, as before.
- **generic one-word** (`codec.md`, `compression.md`) → `WARN`, counted and
  summarized separately, does not fail the scan.

`readme`/`index` remain excluded entirely.

## Why

In glyphive, private plans named `codec.md` and `compression.md` made every
legitimate public API link with those exact filenames fail the scan. A one-word
basename carries almost no signal that the reference is to a *plan* rather than
to a same-named public doc — but the plan-naming rule already asks for
descriptive snake_case names, so a genuinely private plan basename is nearly
always multi-word. The `_` test separates the two cheaply.

Failing on generic names trains the reader to dismiss the tool's output, which
costs more than the false negative: a one-word plan name that really did leak
still surfaces, as a WARN requiring conscious review.

## Note on this repo

`leak_check.py` reports ~347 hits against the dotagents repo itself. Those are
expected: this repo intentionally tracks `.agents/` (the public-repo exception
in its own `AGENTS.md`), so every self-reference matches. `leak_check` is meant
for *consuming* repos; `audit_config.py --repo-hygiene` is the gate here.
