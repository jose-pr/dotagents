---
name: D42
description: cloud-setup wires the recovery hook when DOTAGENTS_AGENTS_REMOTE is unset at setup time too, not just on clone failure, so a secret absent from the setup-script phase but present in-session heals next session
type: decision
---

Symptom (fresh cloud session): `~/.agents` never appeared, the `dotagents` CLI was
never installed, no hooks were wired — same end state D39 was meant to prevent. But
this was NOT the egress-race D39 fixed. Evidence from the setup-only run: the
init-script one-liner was the correct 143-char form, `tools/cloud-setup.sh` was
fetched, and it ran — the setup-only diag log recorded `init_script_completed` with
`output_length: 154` in ~100ms. That length is exactly the banner plus the
`DOTAGENTS_AGENTS_REMOTE unset … skipping` line and nothing else; the clone path
would have logged `cloning …` / `clone attempt N failed …` and slept ~30s across its
5 backoff tries. So the bootstrap took step 2's no-remote `else` branch — which just
`echo`'d and `exit 0`'d — even though the remote/token secrets WERE configured for
the environment (they're set in every interactive session). On this hosted runner the
secrets are exposed to session processes but not injected into the setup-script phase,
so the very first bootstrap has no remote to clone.

Root cause: D39 added the persisted SessionStart recovery hook only on the
clone-*failure* branch. The no-remote branch treated "no remote" as a terminal,
expected condition and left nothing behind to retry — so, unlike a clone failure,
this failure mode could not self-heal, and the environment stayed dead every session.

Fix: the no-remote `else` branch now calls `_dg_install_recovery_hook` before
`exit 0`, same as the exhausted-clone branch. The recovery hook runs at the next
session's start, where the secret IS present, so it clones and (step 5) removes
itself. The script can't distinguish "secret not injected into setup phase yet" from
"remote genuinely never configured", so it treats the ambiguous case optimistically:
a truly remote-less environment just re-skips each session (~100ms; `_dg_settings
present` is idempotent, so the hook never duplicates — verified: hook count stays 1
across repeated runs), while the common hosted case heals on session 2. Verified in a
throwaway `$HOME` with the remote unset: recovery script persisted to
`~/.dotagents/cloud-setup.sh`, one SessionStart hook wired into
`~/.claude/settings.json`, and no clone attempted.

Note (environment side, out of the script's control): the durable fix is also to
expose `DOTAGENTS_AGENTS_REMOTE`/`DOTAGENTS_AGENTS_TOKEN` to the web environment's
Setup Script phase, so the FIRST container succeeds instead of the second. This
decision makes the script robust when that isn't done.
