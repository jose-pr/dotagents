Date: 2026-07-20 (folds findings from 2026-07-12 and 2026-07-16)

## Decision

Two rules that look contradictory but are not:

1. **Never parallelize repo-mutating git commands.** `git add`, `commit`,
   `merge`, `rebase`, `mv` against the same repository are strictly sequential.
2. **Do partition plan execution by write scope.** When asked to execute
   multiple plans or a multi-phase plan, build a phase-by-phase write-scope
   matrix first, then run genuinely independent phases in parallel with explicit
   ownership; serialize only shared files, shared progress, and dependency edges.

## Why they belong together

Both came from over-applying one heuristic. Three logical commits fired in
parallel produced `fatal: Unable to create '.git/index.lock': File exists` — the
"parallelize independent work" guidance was generalized to operations that share
one lock. Conversely, an ordered phase list was treated as a single indivisible
unit and safe parallelism went unused, because shared files were over-weighted
without actually doing the write-scope analysis.

The discriminator is **shared mutable state**, not logical independence. One git
index is shared state however separate the commits are; two plan phases touching
disjoint files are not, however sequential the plan document looks.
