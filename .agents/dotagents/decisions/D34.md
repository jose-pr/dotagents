description: Python version policy (run latest, test floor) + venv naming .venv/<ver>-<os>-<arch> + use Python Manager, avoid Store Python
type: decision
resolves: runtime — Store-Python pip hang, verified 2026-07-14
---

`kb/PYTHON.md` now specifies three linked rules:
- **Version policy**: develop/run on the latest stable Python (currently 3.14) and also
  test the `requires-python` floor (currently 3.9) — catches version-specific breakage
  (e.g. the `Path.write_text(newline=)` 3.10+ gotcha in [[D33]]'s `_wrappers.py`). CI
  runs the full matrix; locally smoke-test latest + floor.
- **Venv naming**: project-local, gitignored `.venv/<ver>-<os>-<arch>[-<task>]` — `<os>`
  ∈ `nt`/`posix`/`darwin`, `<arch>` from `platform.machine().lower()` (e.g.
  `.venv/3.14-nt-amd64`, `.venv/3.14-nt-amd64-docs`). Extends the old flat
  `.venv/<version>` so multiple interpreters/tasks coexist.
- **Interpreter management**: use the **Python Manager** (`py install <ver>`; `py -0p`
  lists real paths); the `py` launcher selects. **Avoid Microsoft Store Python** — its
  app-alias filesystem virtualization makes `pip install` silently hang / no-op (exits 0
  installing nothing). This was the real cause of a "pip suddenly hangs" incident on the
  author's machine; the machine-specific confirmation lives in the private
  `~/.agents/AGENTS.local.md`, the generic rule here.

Verified 2026-07-14: `py -3.14` and `py -3.9` both install the dotagents stack (duho
0.1.1, pathlib_next 0.8.0) and run `dotagents init` correctly; the Store `py -3.12` did
not. `kb/PYTHON.md` is an opt-in example (`payload/examples/kb/`), so this rule ships to
users who install the Python kb, not by default.
