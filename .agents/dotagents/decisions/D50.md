Date: 2026-07-20

## Decision

Any path inside `~/.agents/AGENTS.md` (and the shipped `_overlay/AGENTS.md`,
`flows/*.md`, `kb/*.md`, `MODELS.md`) that points at the global config itself is
written **absolute** with the `~/.agents/` prefix. Only genuinely project-scoped
references (`<project>/...`, `.gitignore` patterns) stay relative.

## Why

Three hosts reach the shared directives by three different mechanisms, and none
of them leaves the file with a usable location of its own:

| Host | Pointer | Mechanism |
| --- | --- | --- |
| Claude Code | `~/.claude/CLAUDE.md` | `@../.agents/AGENTS.md` — content is **inlined** |
| Codex | `~/.codex/AGENTS.md` | NTFS symlink |
| Antigravity | `~/.gemini/config/AGENTS.md` | NTFS symlink |

Inlining resolves a bare relative path against the **session working directory**
— a project dir, where it misses or, worse, hits an unrelated same-named file.
The symlinked hosts are no safer: they open the file at its *link* path, so a
bare `flows/REPO.md` anchors to `~/.codex/flows/` or `~/.gemini/config/flows/`,
neither of which exists. Three hosts, three different wrong answers.

Verified: neither Codex nor Antigravity can be *configured* to point at the
shared file (`model_instructions_file` replaces base instructions wholesale;
`project_doc_fallback_filenames` takes bare filenames, not paths; Antigravity
discovers rules only as files in a directory). The pointer files are the only
mechanism available, so this is not a wiring problem to fix elsewhere.

## Applied

`src/dotagents/_overlay/AGENTS.md`: `flows/REPO.md` → `~/.agents/flows/REPO.md`;
`findings/processed/` → `~/.agents/dotagents/findings/processed/`.

## Note for the installer

Windows symlink creation needs Developer Mode or elevation. A `dotagents`
installer that wires a host pointer must have a copy-based fallback and must not
silently produce a broken link.
