---
name: D41
description: leak_check scans commit messages for agent-session trailers/URLs (Claude-Session:, claude.ai/code/session), not just tracked files, so a harness-added session id can't reach public history unnoticed
type: decision
---

Symptom: 17 commits on public `main` (a full prior session's work) carried a
`Claude-Session: https://claude.ai/code/session_…` trailer. The agent harness appends
that trailer to every commit message automatically, and nothing gated it, so it landed
in a public repo. The session content is account-gated (a stranger with the URL can't
read it), but the session *id* still shouldn't be in public history — and `leak_check.py`
only scanned tracked *files*, never commit messages, so it never flagged this class.

Fix: `tools/leak_check.py` gains `scan_commit_messages()`, run alongside the file scan.
It walks the current branch's history (`git log HEAD`, NUL/`\x1e`-delimited so bodies with
blank lines parse cleanly). To catch real leaks WITHOUT false-positiving on a message that
merely documents the patterns (this tool's own commits, the filter-branch remediation
snippet), it matches the trailer FORM — a line starting with `Claude-Session:` — and an
actual `claude.ai/code/session_<id>` URL, not the bare phrase. Hits feed the same exit-1
total. (First cut matched substrings anywhere and flagged its own commit describing the
check — hence the tighter anchored regexes.) `flows/REPO.md` release discipline now names the check and the remediation
for one that already shipped: `git filter-branch --msg-filter "sed '/^Claude-Session:/d'"`,
with the caveat that pre-rewrite SHAs stay reachable on the host until it GCs (so a rewrite
scrubs branch history but is not a substitute for treating anything *content*-sensitive as
compromised — rotate secrets regardless).

Scope: messages only, current-branch history (not `--all`) — the branch being released is
what matters, and `--all` would drag in stale remote-tracking refs. `Co-Authored-By` is
deliberately NOT flagged: it names the model, not a private id, and is standard OSS
practice. This is the file-leakage rule (D-era Leakage) extended to the commit-message
surface; keeping it in `leak_check` (not a separate tool) means the one pre-release gate
covers both. Verified: 0 hits on the scrubbed history, 2 hits on a planted test commit.
