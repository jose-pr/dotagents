Date: 2026-07-20 (folds two findings from the authkeys 0.4.0 release)

## Decision

The repo standard ships **three** workflow files, not two: `test.yml`,
`release.yml`, and a new `docs.yml`.

- `docs.yml` — triggers on `workflow_dispatch` (optional `ref` input, mirroring
  `test.yml`) **and** `push` to the default branch restricted to docs-relevant
  paths (`docs/**`, `mkdocs.yml`, `README.md`, `CHANGELOG.md`, `src/**`). It
  owns `docs-build` + `docs-deploy`.
- `release.yml` still *builds* docs as a gate (`mkdocs build --strict` catches
  broken docs before publishing) but no longer **deploys** them.
- Add `skip-existing: true` to `pypa/gh-action-pypi-publish` so re-running a
  release for an unrelated job cannot hard-fail on an already-published version.

## Why

`flows/REPO.md`'s own rationale for splitting test from release is *"a release
must never be the first time a config is exercised."* Docs-in-`release.yml`
violated exactly that: the docs/Pages path's first-ever execution was during a
real release. Concrete costs on `authkeys` 0.4.0:

- Docs could only ship by cutting a version — the site sat 404 while the package
  was live on PyPI.
- `deploy-pages` consumes the Pages artifact from the **same run attempt**, so
  `gh run rerun --failed` cannot repair a failed docs deploy. Re-running the
  whole run also re-runs `publish-pypi`, which without `skip-existing` hard-fails
  on the duplicate version — leaving a release permanently red for a docs problem.

Deploying from the default branch also sidesteps the environment policy below
entirely, removing a whole class of first-release failure.

## Pages preconditions are two owner-only calls, not one

`kb/python.md` said "Source: GitHub Actions [is] one-time, owner-only setup."
Necessary but **not sufficient** — a new repo fails twice, for two reasons:

1. Pages not enabled → `actions/configure-pages@v5` fails with *Get Pages site
   failed*. Fix: `gh api repos/<o>/<r>/pages -X POST -f build_type=workflow`
2. Even once enabled, a **tag**-triggered deploy is rejected: *Tag "v0.4.0" is
   not allowed to deploy to github-pages due to environment protection rules.*
   The auto-created `github-pages` environment allows `branch: main` only. Fix:
   `gh api repos/<o>/<r>/environments/github-pages/deployment-branch-policies -X POST -f name='v*' -f type='tag'`

With `docs.yml` deploying from the default branch, (2) becomes unnecessary.

## Debugging note

Job logs for a failed Pages deploy are frequently unavailable via the API
(`BlobNotFound` / 404). The **check-run annotations** are the reliable source:
`gh api repos/<o>/<r>/check-runs/<job_id>/annotations`. Worth reaching for
before burning time on log retrieval for any Actions failure with no readable log.
