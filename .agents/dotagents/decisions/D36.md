---
name: D36
description: the installer is overlay-agnostic — init=base only, install=base + opt-in --overlays <path> (copy only); overlay management deferred to a future subcommand
type: decision
---

The shipped `dotagents` CLI deliberately knows almost nothing about overlays (user
steer: "stay simple, change the API later"):
- `init` = apply the base overlay only.
- `install` = base + any `--overlays <path>` given (repeatable), each just **copied in**
  (create-if-absent, never clobber). No bundled overlays, no `requires` resolution, no
  `AGENTS.md` routing auto-merge. `--from` still selects the base source (URI-capable
  via pathlib_next); `--bin-dir` still writes wrappers.
- Overlays are opt-in **examples** with a pluggable source: the installer bundles none;
  you point `--overlays` at a repo's `overlays/<name>/` (or any overlay dir). A future
  **`dotagents overlays` subcommand** will manage them by name (list/add/remove, resolve
  `requires`, rebuild the AGENTS.md routing region — the manifests already carry
  `requires`/`routing` for it).

Rationale: shipping the full resolve/routing-merge machinery before the API is settled
is premature; a copy-only `--overlays` is enough to layer examples now and cheap to
evolve. The heavier machinery was written then pulled back to `_overlays.py`'s minimal
`apply_overlay` per this decision. Builds on [[D35]]; the future subcommand relates to
the tools-as-subcommands idea (the `tools_as_dotagents_subcommands.md` draft plan).
