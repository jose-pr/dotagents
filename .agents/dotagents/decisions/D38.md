---
name: D38
description: migrate the CLI to duho >= 0.3.3 (Args/Cmd split; commands are (LoggingArgs, Cmd) with __call__) and shim duho's zipapp AST-introspection bug so the .pyz keeps full flags/help
type: decision
---

User steer: "update to latest duho." Latest is 0.3.3 (0.1.1 was pinned). duho's Plan-13
**`Args`/`Cmd` split** is a breaking API change: `LoggingArgs` is now a pure data mixin
with no `__call__`, so `duho.main` rejects a bare-`LoggingArgs` command as "not runnable"
— even `init` failed until migrated (this bit us: the dev container had duho 0.3.3, so
the whole CLI looked broken when it was just the version).

Migration (mechanical, no field changes):
- Leaf commands: `class X(LoggingArgs):` + `__run__` → `class X(LoggingArgs, Cmd):` +
  `__call__` (data mixin first, executable base last, per duho's recommended MRO).
- Umbrella: `Dotagents(LoggingArgs, Cli)` — `Cli` is the app-root layer that owns
  `_version_`/`_subcommands_`/completion. `_subcommands_`, `_parsername_`, and the
  annotation + help-string + flags-tuple field syntax are all unchanged.
- `pyproject` `duho>=0.1.1`→`>=0.3.3`; `build-pyz` vendored default `0.1.1`→`0.3.3`.
  duho 0.3.3 is `requires-python >=3.9`, so the 3.9 floor holds.

Zipapp regression + shim: duho 0.3.3 discovers each field's flags/help by AST-parsing its
module source (`_introspect.getclsdef` → `Path(module.__file__).read_text()`). In a
`.pyz` `__file__` is a zip-internal path `read_text()` can't open, and `getclsdef`
catches that `OSError` **before** its `inspect.getsource` fallback — so every field loses
its declared flags + help: the `link` positional degraded to `--path`, `--from` to
`--from-`, help text vanished. This surfaced now because `link` added the CLI's first
positional (name-derivation had masked it for options). Since the downloadable `.pyz` is
a featured install mode, `cli.main` calls `_repoint_zipapp_sources()`: for
`dotagents.cli` + `duho.presets` (the only field-bearing sources dispatched), if
`__file__` isn't a real path, extract the source to a temp file and repoint `__file__`
so duho's read succeeds. No-op for a plain install. ~25 lines, self-contained, and
harmless if a future duho removes the bug (it just repoints `__file__`). A `build-pyz`
CI smoke asserting the `link <path>` positional + `--from` guards against silent
re-breakage. Upstream-fixable in duho (try `getsource` even when `_module_index` raises);
tracked here so the shim can be dropped when duho fixes it.
